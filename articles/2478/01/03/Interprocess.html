Linux Control Group - No Internal Process Constraint

January 3, 2478

Let’s create the following cgroup hierarchy. Notice how we enable the resource controller +pids on testing/’s and cg1/’s subtrees but not on cg2/’s.

$ cd /sys/fs/cgroup $ mkdir -p testing/cg1/cg1_1 $ mkdir -p testing/cg1/cg1_2 $ mkdir -p testing/cg2/cg2_1 $ mkdir -p testing/cg2/cg2_2 $ cd testing/ $ echo '+pids' > cgroup.subtree_control $ echo '+pids' > cg1/cgroup.subtree_control

Let’s create a long running process and let’s move it to one cgroup. In fact, let’s try different cgroups.

$ sleep 1000 &
[<job-id-proc1>] <pid-proc1>

$ echo <pid-proc1> > cgroup.procs            # byexample: +paste
$ echo <pid-proc1> > ../cgroup.procs         # byexample: +paste

$ echo <pid-proc1> > cg1/cgroup.procs        # byexample: +paste
$ echo <pid-proc1> > ../cgroup.procs         # byexample: +paste

$ echo <pid-proc1> > cg1/cg1_1/cgroup.procs        # byexample: +paste
$ echo <pid-proc1> > ../cgroup.procs               # byexample: +paste

$ echo <pid-proc1> > cg2/cg2_1/cgroup.procs        # byexample: +paste
$ echo <pid-proc1> > ../cgroup.procs               # byexample: +paste

The process could be moved to testing/, cg1/, cg1_1 and cg2_1 without problems.

But you may wonder, what is the need of echo <pid-proc1> > ../cgroup.procs1, why after moving the process to a cgroup I had to move it to the root before move it again to another cgroup?

The hierarchy is empty so the resource controller is not acting on any process yet. Once a process is added to one of the cgroups the “no internal process” constraint enters in action.

Let me show you. I will move the sleep process to cg1 and then move it again to cg1_1 as before but without moving it to the root in between.

$ echo <pid-proc1> > cg1/cgroup.procs        # byexample: +paste

$ echo <pid-proc1> > cg1/cg1_1/cgroup.procs        # byexample: +paste
<...>write error: Operation not supported

Okay, it seems that we cannot move our process after all!

Is because we cannot move it from where it is or because we cannot move it to where we want?

We could move it outside so it is likely that the problem is that we cannot move it to cg1/cg1_1.

Let’s try with another process:

$ sleep 1000 &
[<job-id-proc2>] <pid-proc2>

$ echo <pid-proc2> > cg1/cg1_1/cgroup.procs        # byexample: +paste
<...>write error: Operation not supported

$ echo <pid-proc2> > cg1/cg1_2/cgroup.procs        # byexample: +paste
<...>write error: Operation not supported

$ echo <pid-proc2> > cgroup.procs                  # byexample: +paste
<...>write error: Device or resource busy

$ echo <pid-proc2> > cg1/cgroup.procs              # byexample: +paste

Interesting… so far we have:

  • if the hierarchy is empty you can add a process at any level
  • if it is not empty, adding a process into a subtree fails with an Operation not supported
  • if it is not empty, adding a process into a parent cgroup fails with a Device or resource busy. An exception of this is the root cgroup.
  • if it is not empty, adding a process into the same cgroup that already has processes succeeds.
